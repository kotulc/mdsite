# nlp-mdsite

A Next.js static site template for intelligent, agent-friendly publishing from markdown.

Point it at a directory of markdown files, run a single command, and get a fully built
static website ready for GitHub Pages.


## Purpose

`nlp-mdsite` provides a minimal, mobile-first site template designed to work at the end of a
markdown publishing pipeline. The goal is a push-button workflow: point the pipeline at a
directory of markdown files, run a single command, and get a fully built, deployed website.   

The template is designed to be configured as easily by an AI agent as by a human вЂ” a small
config change is all that's required to launch a new site.


## Getting Started

## Features

- **Markdown в†’ MDX** вЂ” automatic conversion, any folder structure
- **Images** вЂ” copied and path-rewritten automatically; corrupt EXIF data stripped
- **Reading time** вЂ” estimated and injected into every page's frontmatter
- **Tags and categories** вЂ” rendered as pill chips below each title and in the sidebar
- **Sidebar metrics** вЂ” any numeric frontmatter field surfaces as a labeled score
- **Nav ordering** вЂ” configure page and folder order via `site.config.js` or frontmatter `order:`
- **Per-page feed** вЂ” scroll to the bottom of any page to load the next one inline
- **Theme toggle** вЂ” light / dark / system toggle in the navbar
- **GitHub header icon** вЂ” circular GitHub repo link, auto-shown from `repo_url` in config
- **GitHub Pages** вЂ” one workflow, zero manual steps


## Intelligence Layer

The "NLP" in `nlp-mdsite` refers to a set of built-in and planned intelligent features:

- **Rich metadata display** вЂ” tags, reading metrics, and other signals generated by the upstream
  pipeline are displayed as first-class UI elements on each page
- **Semantic search** *(planned)* вЂ” search across all content by meaning, not just keywords
- **Semantic theming** *(planned)* вЂ” the site's visual identity is automatically derived from
  content signals rather than manually specified

These features are intended as defaults, not plugins. The upstream pipeline (`mdpub`) is
responsible for generating the metadata and semantic signals; this template consumes and
renders them.


## How It Works

1. Write content as markdown files in any folder structure
2. Run `npm run ingest` to mirror content into `pages/`, rewrite paths, and generate indexes
3. Run `npm run build` to produce the static site in `out/`
4. Deploy to GitHub Pages (or any static host) вЂ” included workflow does this automatically


## Getting Started

### Prerequisites

- [Node.js](https://nodejs.org/) 18 or later
- npm (bundled with Node.js)

### Setup

**1. Install dependencies**
```bash
npm install
```

**2. Add your content**

Place your markdown files in a source directory in any folder structure. The directory tree
is mirrored directly into the site вЂ” subdirectories become URL path segments.

```
<source>/
в”њв”Ђв”Ђ home.md          # home.md or index.md at any level в†’ section landing page
в”њв”Ђв”Ђ about-us.md
в”њв”Ђв”Ђ images/          # images/ next to any .md files в†’ copied to public/images/
в””в”Ђв”Ђ posts/           # any subdirectory structure is valid
    в””в”Ђв”Ђ <year>/
        в”њв”Ђв”Ђ post-slug.md
        в””в”Ђв”Ђ images/
```

Frontmatter fields used by the template: `title`, `date`, `order`, `categories`, `tags`.

**3. Configure the site**

Edit [site.config.js](site.config.js) to set the site title and base URL:
```js
module.exports = {
  title: 'My Site',
  base_url: 'https://username.github.io',
  base_path: '/repo-name',   // empty string for root domains
  repo_url: 'https://github.com/username/repo-name',
}
```

**4. Ingest content**
```bash
npm run ingest docs            # default: ingest from docs/
npm run ingest path/to/source  # ingest from a custom directory
```

This mirrors markdown into `pages/`, renames `.md в†’ .mdx`, copies images to `public/images/`,
rewrites image and link paths, injects reading time, and generates `_meta.json` ordering files,
`posts-index.json`, and `feed-index.json`. It also syncs `README.md` to the `/about` page.

**5. Preview locally**
```bash
npm run dev        # development server with hot reload
npm run build      # production build в†’ out/
npm run preview    # serve the built out/ directory
```


## Publishing to GitHub Pages

The included workflow (`.github/workflows/deploy.yml`) builds and deploys automatically
on every push to `main`. Follow these steps to enable it for the first time:

**1. Push to GitHub**
```bash
git add .
git commit -m "Add docs content and configure deployment"
git push
```

This first push creates the `gh-pages` branch automatically.

**2. Enable GitHub Pages** *(one-time setup вЂ” do this after the first push)*

Go to your repository on GitHub:
- **Settings в†’ Pages в†’ Build and deployment**
- Set **Source** to `Deploy from a branch`
- Set **Branch** to `gh-pages`, folder `/ (root)`
- Click **Save**

**3. Set the BASE_PATH variable** *(project pages repos only)*

If your site lives at a subpath (e.g. `username.github.io/repo-name`), set this variable:
- **Settings в†’ Secrets and variables в†’ Actions в†’ Variables в†’ New repository variable**
- Name: `BASE_PATH`, Value: `/repo-name` (e.g. `/nlp-mdsite`)

Skip this step if you are deploying to a root domain (`username.github.io`).

**4. Watch the workflow**

Go to the **Actions** tab on GitHub. The "Deploy to GitHub Pages" workflow should be running
(or complete). A green checkmark means the deploy succeeded.

**5. Access the live site**

Your site will be live at:
```
https://<username>.github.io/<repo-name>/
```
GitHub Pages can take 1вЂ“2 minutes to propagate after the first deploy.


## Using as a Reusable Workflow

Any GitHub repository can publish its docs folder to GitHub Pages using mdsite as the build
engine вЂ” no cloning or forking required. The deploy workflow supports `workflow_call` so other
repos can call it directly.

### Setup in the calling repo

**1. Create a site config**

Add `site.config.js` to the root of your repo. The workflow will copy it over mdsite's defaults
before building:

```js
module.exports = {
  title: 'My Project',
  base_url: 'https://username.github.io',
  base_path: '/repo-name',
  repo_url: 'https://github.com/username/repo-name',
}
```

**2. Add the caller workflow**

Create `.github/workflows/publish-docs.yml`:

```yaml
name: Publish Docs

on:
  push:
    branches: [main]
    paths: ['docs/**', 'site.config.js']
  workflow_dispatch:

jobs:
  publish:
    uses: kotulc/mdsite/.github/workflows/deploy.yml@main
    permissions:
      contents: write
    with:
      content_repo: owner/repo-name
      content_path: docs
      base_path: /repo-name
```

Replace `owner/repo-name` with your GitHub repository slug and `/repo-name` with the
subpath where your GitHub Pages site lives.

**3. Trigger the first deploy**

Push to `main` (or trigger manually via **Actions в†’ Publish Docs в†’ Run workflow**). This
creates the `gh-pages` branch in your repo.

**4. Enable GitHub Pages** *(one-time setup вЂ” do this after the first push)*

In your repo on GitHub:
- **Settings в†’ Pages в†’ Build and deployment**
- Set **Source** to `Deploy from a branch`
- Set **Branch** to `gh-pages`, folder `/ (root)`

All subsequent pushes to `docs/` or `site.config.js` deploy automatically from here.

### How it works

The reusable workflow:
1. Checks out `kotulc/mdsite` for the build tooling
2. Checks out your repo into `_content/`
3. Copies your `site.config.js` if present
4. Ingests your docs, builds, and pushes to your repo's `gh-pages` branch

`GITHUB_TOKEN` in a called workflow resolves to the **calling repo's** token, so the deploy
always targets your repo's `gh-pages` вЂ” not mdsite's.


## Integrations

`nlp-mdsite` is the rendering and build step at the end of the [`mdpub`](https://github.com/kotulc/nlp-mdpub)
publishing pipeline. Configuration, API interfaces, and MCP tooling live in `mdpub`;
this template is driven entirely by its config file and content directory.
